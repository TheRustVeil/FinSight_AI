## 0. Provision data stores (once)

1. **Neon** вҶ’ create a Postgres database вҶ’ copy the connection string (`DATABASE_URL`).
2. **Upstash** вҶ’ create a Redis database вҶ’ copy the `rediss://` URL (`REDIS_URL`).
3. **Google AI Studio** вҶ’ create an API key (`GEMINI_API_KEY`) for the AI advisor.

---

## 1. Backend вҶ’ Render (`apps/api`)

The repo includes [`render.yaml`](render.yaml), so Render can provision everything from a Blueprint.

### Why Render (not Vercel) for the API?
The API is a **long-running Express server** that also runs **BullMQ workers in-process** (CSV imports, insights cron). That doesn't fit a serverless model вҖ” it needs a persistent Node process.

### Steps
1. Push this repo to GitHub (done).
2. Render вҶ’ **New +** вҶ’ **Blueprint** вҶ’ select the repo. It reads `render.yaml`.
3. Fill in the `sync:false` env vars in the dashboard:
   | Var | Value |
   |-----|-------|
   | `DATABASE_URL` | Neon connection string |
   | `REDIS_URL` | Upstash `rediss://вҖҰ` URL |
   | `GEMINI_API_KEY` | Google AI Studio key |
   | `FRONTEND_URL` | **REQUIRED** вҖ” your exact Vercel origin, e.g. `https://fin-sight-ai-web.vercel.app` (https, no trailing slash). This is the CORS allow-origin; if it's wrong/unset, the browser blocks all auth requests. |
   | `GOOGLE_*` / `SMTP_*` | optional (OAuth / email) |

   `JWT_ACCESS_SECRET` and `JWT_REFRESH_SECRET` are auto-generated by Render.
   `AUTH_AUTO_VERIFY=true` is preset in `render.yaml` so signups work without an SMTP server (see the email note below).
4. Deploy. Render runs `corepack enable && yarn install` (which runs `prisma generate` via `postinstall`) and starts `yarn workspace api start` вҶ’ `tsx src/server.ts`.
5. **Initialize the database** once вҖ” from the Render Shell (or locally with the prod `DATABASE_URL`):
   ```bash
   yarn workspace api db:push     # create tables
   yarn workspace api db:seed     # seed 18 categories
   ```
6. Health check: `https://<your-api>.onrender.com/health` вҶ’ `{ "status": "ok" }`.

> **Free-plan note:** Render free services sleep when idle (cold starts ~30s) and the in-process insights cron won't fire while asleep. Fine for a demo; upgrade for always-on workers.

### Keep the free API awake (optional but recommended)
To avoid cold starts and keep the in-process workers/cron alive on the free plan, ping the health endpoint every ~10 minutes with a free uptime monitor:
1. Create a free monitor at [UptimeRobot](https://uptimerobot.com) or [cron-job.org](https://cron-job.org).
2. **URL:** `https://<your-api>.onrender.com/health` В· **Interval:** 5вҖ“10 min.

One always-on free service uses ~730 of Render's 750 free instance-hours/month, so this fits within the free allowance.

## 2. Frontend вҶ’ Vercel (`apps/web`)

1. Vercel вҶ’ **Add NewвҖҰ вҶ’ Project** вҶ’ import the repo.
2. **Root Directory** вҶ’ `apps/web`.
3. Framework preset: **Next.js** (auto-detected). Leave build/install commands default вҖ” Vercel handles the Yarn monorepo.
4. **Environment Variables:**
   | Var | Value |
   |-----|-------|
   | `NEXT_PUBLIC_API_URL` | your Render API URL, e.g. `https://finsight-api.onrender.com` |
5. Deploy. Visit the Vercel URL.
6. Go back to Render and set `FRONTEND_URL` (and `GOOGLE_CALLBACK_URL` if using OAuth) to the Vercel domain, then redeploy the API.

> `NEXT_PUBLIC_API_URL` is read at **build time** by `next.config.ts` (the rewrite). If you change it, **redeploy** the web app.

> вҡ пёҸ **CORS gotcha (most common deploy failure).** The web client calls the Render API cross-origin, so Render's `FRONTEND_URL` **must exactly match** the Vercel origin вҖ” `https`, **no trailing slash**, no path. If it's left at the `http://localhost:3000` default, every register/login fails with `Access-Control-Allow-Origin ... not equal to the supplied origin` and `net::ERR_FAILED` in the console. Fix: set `FRONTEND_URL` on Render вҶ’ redeploy.

---

## 3. Post-deploy checklist

- [ ] `GET https://<api>/health` returns `{ status: "ok" }`
- [ ] Database tables created + categories seeded (`db:push` + `db:seed`)
- [ ] `NEXT_PUBLIC_API_URL` (Vercel) points at the Render API
- [ ] `FRONTEND_URL` (Render) **exactly** matches the Vercel origin вҖ” `https`, no trailing slash (this is the CORS allow-origin)
- [ ] Register a user on the live site вҶ’ login works (cookie set via the proxy)
- [ ] (If OAuth) Google console has `https://<api>/api/v1/auth/google/callback` as an authorized redirect URI

---

## Notes & gotchas

- **Secrets:** only `.env.example` / `.env.local.example` are committed. Set real secrets in each platform's dashboard. Never commit `.env`.
- **Email verification:** in production, signup normally sends a verification email and login is blocked until verified. Dev (MailHog) won't work on Render, so `render.yaml` sets `AUTH_AUTO_VERIFY=true` to auto-verify accounts. To enable real emails instead, configure a real SMTP provider (`SMTP_HOST/PORT/USER/PASS`) and remove `AUTH_AUTO_VERIFY`.
- **`tsx` in production:** the API starts with `tsx` (not a compiled `dist/`) because the `shared` workspace package exports TypeScript directly. `tsx` and `prisma` are therefore runtime `dependencies`, not devDependencies.
- **Neon cold start:** the first DB request after idle can time out while the compute wakes (~a few seconds); retry once. `/health` doesn't touch the DB, so it stays green even while the DB sleeps.
- **`--frozen-lockfile` fails on Render?** Use `yarn install` (without the flag) as the build command.
- **File uploads:** CSV uploads are written to local disk, which is ephemeral on Render. For durable storage, configure the `AWS_S3_*` env vars (the import module supports S3-compatible storage).
